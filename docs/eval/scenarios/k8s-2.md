# Scenario: k8s-2 — crashloop-backoff

**Layer:** k8s
**Root-cause layer:** k8s
**Correct action class:** apply_patch

## What This Scenario Tests

This scenario models a crash-loop caused by injecting the environment variable `APP_CRASH_MODE=1` into `Deployment/vigil-app`. The root-cause component is `Deployment/vigil-app container crashes on APP_CRASH_MODE=1 env var`. The container starts, reads the env var, and immediately exits, producing a `CrashLoopBackOff` event.

## Why This Scenario Was Chosen

Exercises the `apply_patch` path for in-spec environment variable fixes. Unlike k8s-1 (image regression), the pod image is valid — the fault is a misconfigured environment variable that causes the application to crash on startup. The agent must read pod events to identify the offending env var and remove it via a JSON patch.

## Inject / Reset Commands

**Inject:**
```bash
./eval/scenarios/k8s-2/inject.sh <seed>
```

Patches `Deployment/vigil-app` to add `APP_CRASH_MODE=1` as an environment variable via a JSON patch operation, causing the container to crash on every start.

**Reset:**
```bash
./eval/scenarios/k8s-2/reset.sh <seed>
```

Removes the `APP_CRASH_MODE` env var via a JSON patch `remove` operation, re-applies the deployment manifest from `manifests/`, waits for rollout, and resumes the `flux-system` kustomization.

## Expected Agent Reasoning Path

Steps from `expected_resolution_path: "diagnosis -> flux_suspend -> apply_patch (remove env) -> watchdog_confirm -> flux_resume"`:

1. **Diagnosis layer:** kubectl-mcp reports `CrashLoopBackOff` on `Deployment/vigil-app`; pod logs or events indicate `APP_CRASH_MODE=1` is set
2. **Escalation decision:** `requires_os_level=False` — fault is entirely within the Kubernetes layer; no OS escalation
3. **Remediation:** `apply_patch` to remove the `APP_CRASH_MODE` environment variable from the container spec after `flux_suspend`; Flux is resumed after pods stabilise
4. **Watchdog:** all pods in `vigil-app` deployment return to Running; `CrashLoopBackOff` events cease

## Success Criteria

- Agent identifies root-cause layer correctly (k8s vs os)
- Agent escalates to OS layer if and only if root_cause_layer=os
- Remediation matches correct_action_class
- Watchdog confirms recovery (no destructive_repair=true in run record)

## Known Edge Cases

- The agent must identify the specific env var (`APP_CRASH_MODE`) rather than issuing a generic rollback; `rollout_undo` would also restore baseline but the correct action class is `apply_patch`
- Flux must be suspended before patching; Flux will re-add the env var if reconciliation runs during remediation
- reset.sh applies both a JSON patch and the full manifest; this double-application is safe and ensures the deployment is fully restored regardless of state
