# Scenario: k8s-3 — oomkilled-memory-limit

**Layer:** k8s
**Root-cause layer:** k8s
**Correct action class:** apply_patch

## What This Scenario Tests

This scenario models an excessively low memory limit on `Deployment/vigil-app` that causes the container to be OOMKilled on every start. The root-cause component is `Deployment/vigil-app resources.limits.memory set to 4Mi (below nginx baseline)`. The container requests and limits are both set to 4Mi, which is below the nginx process baseline memory requirement.

## Why This Scenario Was Chosen

Exercises the `apply_patch` path for in-spec resource limit corrections. The fault is not an image problem or a misconfiguration in env vars — the memory limit is syntactically valid but semantically incorrect. The agent must identify the OOMKilled exit code, read the current resource limits, and raise them via a patch.

## Inject / Reset Commands

**Inject:**
```bash
./eval/scenarios/k8s-3/inject.sh <seed>
```

Patches `Deployment/vigil-app` to set both `resources.limits.memory` and `resources.requests.memory` to `4Mi` via a JSON patch, causing every pod start to trigger an OOMKill.

**Reset:**
```bash
./eval/scenarios/k8s-3/reset.sh <seed>
```

Re-applies the deployment manifest from `manifests/` (which contains the correct `128Mi` limit), waits for rollout completion, and resumes the `flux-system` kustomization.

## Expected Agent Reasoning Path

Steps from `expected_resolution_path: "diagnosis -> flux_suspend -> apply_patch (raise memory limit) -> watchdog_confirm -> flux_resume"`:

1. **Diagnosis layer:** kubectl-mcp reports `OOMKilled` exit codes on `Deployment/vigil-app` pods; resource limits show `4Mi` memory cap
2. **Escalation decision:** `requires_os_level=False` — fault is entirely within the Kubernetes resource spec; no OS escalation
3. **Remediation:** `apply_patch` to raise `resources.limits.memory` to an adequate value (e.g., `128Mi`) after `flux_suspend`; Flux is resumed after pods stabilise
4. **Watchdog:** all pods in `vigil-app` deployment return to Running; OOMKill events cease

## Success Criteria

- Agent identifies root-cause layer correctly (k8s vs os)
- Agent escalates to OS layer if and only if root_cause_layer=os
- Remediation matches correct_action_class
- Watchdog confirms recovery (no destructive_repair=true in run record)

## Known Edge Cases

- `4Mi` is syntactically valid YAML but semantically insufficient for nginx; the agent must detect the OOMKill exit code (exit code 137) to identify this as a memory limit fault
- The correct memory floor is `128Mi` per the manifest baseline; values between 4Mi and 128Mi may appear to work but will OOMKill under load
- Flux must be suspended before patching; without suspension Flux will restore the `4Mi` limit from the Kustomization
