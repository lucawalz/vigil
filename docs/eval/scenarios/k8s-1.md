# Scenario: k8s-1 — wrong-image-tag

**Layer:** k8s
**Root-cause layer:** k8s
**Correct action class:** rollout_undo

## What This Scenario Tests

This scenario models a bad image tag pushed to `Deployment/vigil-app`, causing the container to enter `ImagePullBackOff`. The root-cause component is `Deployment/vigil-app image nginx:bad-tag-v9 (ImagePullBackOff)`. All pods on the deployment fail to start; the cluster is otherwise healthy and nodes remain Ready.

## Why This Scenario Was Chosen

Exercises the `rollout_undo` path for image regression — the most common real-world Kubernetes incident pattern. Validates that the agent identifies the bad image tag from pod events, suspends Flux to prevent reconciliation interference, and issues `rollout_undo` rather than applying a manual patch.

## Inject / Reset Commands

**Inject:**
```bash
./eval/scenarios/k8s-1/inject.sh <seed>
```

Sets the `vigil-app` container image to `nginx:bad-tag-v9` via `kubectl set image`, triggering `ImagePullBackOff` on all pods.

**Reset:**
```bash
./eval/scenarios/k8s-1/reset.sh <seed>
```

Sets the image back to `nginx:stable`, waits for rollout completion, and resumes the `flux-system` kustomization.

## Expected Agent Reasoning Path

Steps from `expected_resolution_path: "diagnosis -> flux_suspend -> rollout_undo or apply_patch -> watchdog_confirm -> flux_resume"`:

1. **Diagnosis layer:** kubectl-mcp reports `ImagePullBackOff` events on `Deployment/vigil-app`; image tag `nginx:bad-tag-v9` identified as the fault
2. **Escalation decision:** `requires_os_level=False` — fault is entirely within the Kubernetes layer; no OS escalation
3. **Remediation:** `rollout_undo` via `kubectl rollout undo deployment/vigil-app` after `flux_suspend`; Flux is resumed after pods return to Running
4. **Watchdog:** all pods in `vigil-app` deployment return to Running; `ImagePullBackOff` events cease

## Success Criteria

- Agent identifies root-cause layer correctly (k8s vs os)
- Agent escalates to OS layer if and only if root_cause_layer=os
- Remediation matches correct_action_class
- Watchdog confirms recovery (no destructive_repair=true in run record)

## Known Edge Cases

- The agent must suspend Flux before issuing `rollout_undo`; Flux will overwrite the rollback if not suspended, causing a reconciliation loop
- `rollout_undo` and `apply_patch` both appear in the resolution path; either is valid. `rollout_undo` is the canonical correct action class
- reset.sh uses `kubectl set image` to restore `nginx:stable` rather than re-applying the manifest, which is faster and avoids a full Flux reconcile cycle
