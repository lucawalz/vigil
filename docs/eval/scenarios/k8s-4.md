# Scenario: k8s-4 — missing-configmap

**Layer:** k8s
**Root-cause layer:** k8s
**Correct action class:** apply_patch

## What This Scenario Tests

This scenario models a deleted ConfigMap that the `vigil-app` deployment relies on for a volume mount. The root-cause component is `ConfigMap/vigil-app-config missing; deployment cannot mount volume`. After the ConfigMap is deleted and the deployment is restarted, pods fail to start because the volume mount cannot be satisfied.

## Why This Scenario Was Chosen

Exercises the `apply_patch` path for dependency resource restoration. Unlike the image or env-var scenarios, this fault is in a separate resource (ConfigMap) rather than the Deployment spec itself. The agent must identify the missing volume dependency and re-create the ConfigMap from the known manifest.

## Inject / Reset Commands

**Inject:**
```bash
./eval/scenarios/k8s-4/inject.sh <seed>
```

Deletes `ConfigMap/vigil-app-config` from the `default` namespace, then restarts the `vigil-app` deployment to surface the mount failure. Pods will fail with a `CreateContainerConfigError` or equivalent volume mount error.

**Reset:**
```bash
./eval/scenarios/k8s-4/reset.sh <seed>
```

Re-applies the full manifest set from `manifests/` (which re-creates the ConfigMap), restarts the deployment, waits for rollout completion, and resumes the `flux-system` kustomization.

## Expected Agent Reasoning Path

Steps from `expected_resolution_path: "diagnosis -> flux_suspend -> apply_patch (recreate configmap from manifest) -> watchdog_confirm -> flux_resume"`:

1. **Diagnosis layer:** kubectl-mcp reports pods in `Pending` or `ContainerCreating` with volume mount errors; `ConfigMap/vigil-app-config` is missing from the namespace
2. **Escalation decision:** `requires_os_level=False` — fault is entirely within the Kubernetes resource state; no OS escalation
3. **Remediation:** `apply_patch` to re-create `ConfigMap/vigil-app-config` from the manifest after `flux_suspend`; deployment is restarted; Flux is resumed after pods stabilise
4. **Watchdog:** all pods in `vigil-app` deployment return to Running; volume mount errors cease

## Success Criteria

- Agent identifies root-cause layer correctly (k8s vs os)
- Agent escalates to OS layer if and only if root_cause_layer=os
- Remediation matches correct_action_class
- Watchdog confirms recovery (no destructive_repair=true in run record)

## Known Edge Cases

- The agent must identify the missing ConfigMap as the root cause before suspending Flux; without this identification a generic restart will continue to fail
- Flux would normally restore the ConfigMap on the next reconcile cycle, but Flux is suspended during the eval run; the agent must recreate it explicitly
- After re-creating the ConfigMap, the deployment must be restarted to mount the new volume; pods already in a failed state do not automatically retry
