# Scenario: k8s-5 — pvc-pending-bad-storageclass

**Layer:** k8s
**Root-cause layer:** k8s
**Correct action class:** apply_patch

## What This Scenario Tests

This scenario models a `PersistentVolumeClaim` that references a non-existent StorageClass, preventing the claim from binding and leaving the `vigil-app` deployment stuck in `Pending`. The root-cause component is `PersistentVolumeClaim/vigil-app-data references nonexistent StorageClass 'does-not-exist'`. The deployment is scaled to zero to allow PVC deletion and re-creation.

## Why This Scenario Was Chosen

Exercises the `apply_patch` path for PVC recreation under a storage class fault. PVCs are immutable once bound — fixing a bad StorageClass reference requires deleting and re-creating the PVC, then scaling the deployment back up. This tests the agent's ability to handle immutable resource replacement.

## Inject / Reset Commands

**Inject:**
```bash
./eval/scenarios/k8s-5/inject.sh <seed>
```

Scales `vigil-app` to zero replicas, waits for pod deletion, deletes the existing PVC `vigil-app-data`, then creates a new PVC referencing the nonexistent StorageClass `does-not-exist`. The deployment remains scaled to zero with the fault PVC in `Pending`.

**Reset:**
```bash
./eval/scenarios/k8s-5/reset.sh <seed>
```

Scales `vigil-app` to zero, deletes the fault PVC, re-applies the full manifest set (which recreates the PVC with `local-path`), scales back to 1 replica, waits for rollout completion (up to 180s), and resumes the `flux-system` kustomization.

## Expected Agent Reasoning Path

Steps from `expected_resolution_path: "diagnosis -> flux_suspend -> apply_patch (recreate PVC with local-path) -> watchdog_confirm -> flux_resume"`:

1. **Diagnosis layer:** kubectl-mcp reports `Deployment/vigil-app` pods in `Pending`; `PVC/vigil-app-data` shows status `Pending` with reason `no persistent volumes available for this claim`; StorageClass `does-not-exist` is not registered
2. **Escalation decision:** `requires_os_level=False` — fault is entirely within the Kubernetes storage layer; no OS escalation
3. **Remediation:** `apply_patch` to delete the fault PVC and re-create it with StorageClass `local-path` after `flux_suspend`; deployment is scaled back up; Flux is resumed after pods stabilise
4. **Watchdog:** PVC binds to a PersistentVolume; all pods in `vigil-app` deployment return to Running

## Success Criteria

- Agent identifies root-cause layer correctly (k8s vs os)
- Agent escalates to OS layer if and only if root_cause_layer=os
- Remediation matches correct_action_class
- Watchdog confirms recovery (no destructive_repair=true in run record)

## Known Edge Cases

- PVCs cannot be patched in-place to change the StorageClass; the agent must delete the PVC and re-create it; a naive `apply_patch` on the existing PVC will be rejected by the API server
- PVC deletion is blocked until the deployment is scaled to zero and all pods have terminated; reset.sh uses `--wait=true --timeout=60s` on the delete; the agent must follow the same sequence
- reset.sh allows 180s for rollout (compared to 120s in other scenarios) because PVC binding and volume mount add startup latency
